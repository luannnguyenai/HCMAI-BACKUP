# Proposal 04 - Fine-tuning Plan

> Pre-trained models are great but we will get ‾5-15% recall@1 lift by fine-tuning on competition-style data. This proposal describes the cheap-and-fast plan.

## 1. Philosophy

- **LoRA only**. No full-parameter training, ever.
- **Synthetic data** generated by frontier VLMs. No human labelling effort.
- **Vietnamese-first** captions and queries.
- **One model at a time**. Don't bundle changes; A/B vs baseline always.

## 2. Three fine-tuning interventions, in priority order

### 2.1 DreamLIP-style synthetic-caption fine-tuning of SigLIP-2

**Goal**: lift Vietnamese text->image retrieval on competition data.

**Method**:
1. Sample 50K keyframes from the AIC dataset (stratified by place_label / adl_label).
2. For each keyframe, generate **5 long Vietnamese captions** with Qwen2.5-VL-72B INT4:
   - 1 dense caption (50-100 words)
   - 2 short captions (10-20 words)
   - 2 question-style captions ("What is in this image?")
3. **Sub-caption sampling**: at training time randomly drop sentences from the dense caption to form variable-length queries (DreamLIP recipe).
4. **Grouping loss**: positive pairs include (anchor, dense), (anchor, short), (anchor, question). Hard negatives mined from top-50 SigLIP-2 nearest neighbours of the anchor.
5. **LoRA r=16, alpha=32** on Q,K,V,O of both image and text towers.
6. Train 5 epochs on 50K x 5 = 250K pairs, batch 256, LR 1e-4 cosine. Total: ‾6 GPU-hours on A6000.

**Expected lift**: +3-7% R@1 on internal dev set. (DreamLIP paper shows 30M pairs matches CLIP-400M; our 250K is enough for a domain shift.)

**Files**:
- `train/dreamlip_siglip2_lora.py`
- `prompts/caption_generation.txt`

### 2.2 ColVintern late-interaction head for Vietnamese document/OCR pages

**Goal**: lift OCR/transcript retrieval for queries involving Vietnamese named entities and small printed text.

**Method**:
1. Take frames with rich OCR (>= 20 words detected).
2. Use **5CD-AI/ColVintern-1B-v1** as base.
3. Fine-tune the projection head only with **MaxSim contrastive loss** on (OCR-text-query, frame-image) pairs.
4. Queries generated from OCR text via Vintern-3B-beta: "Sinh ra mo^.t ca^u truy va^?n tu+. nhien tie^?ng Vie^.t ma` ngu+o+`i du`ng se‾ go‾ de^? ti`m ki^?m hi`nh a?nh co' va(n ba?n na`y: '<ocr>'"
5. Train 3 epochs on ‾30K pairs. ‾3 GPU-hours.

**Expected lift**: +2-4% recall on OCR-heavy queries; bigger lift on document-style frames (slides, signage).

### 2.3 Cross-encoder reranker on Vietnamese queries

**Goal**: replace the off-the-shelf BLIP-2 ITM with a Vietnamese-aware cross-encoder.

**Method**:
1. Take 100K (query, frame) pairs from our internal mock-task practice + DreamLIP synthetic captions.
2. Label positives (matching ground truth or top-3 according to ground truth) vs negatives (sampled from hard-negative pool: top-50 SigLIP-2 NN minus the positive set).
3. Architecture: Vintern-1B-v3.5 backbone + a classification head returning logit -> sigmoid.
4. LoRA r=8 on the language and vision adapters; freeze backbone.
5. Train BCE loss, 2 epochs, batch 32, ‾4 GPU-hours.

**Expected lift**: +5-10% NDCG@10 on the dev set vs no-rerank.

## 3. Data generation budget

| Asset | Volume | Cost |
|---|---|---|
| Captions (Qwen-VL-72B INT4 local) | 250K @ ‾1500 tokens out | 0 (own GPU, ‾24 hours on A6000) |
| Cross-encoder labels | 100K pairs | 0 (auto from dev set) |
| Eval queries | 500 | Manual; 1 weekend of team time |

If we lack the GPU for local 72B, fallback to **Gemini 2.5 Flash** at $0.50/M in + $3.00/M out:
- 250K captions x avg (200 in + 1500 out) tokens = 50M in + 375M out
- Cost: $25 in + $1125 out = **‾$1150**

Acceptable one-off cost. If budget is tight, do 50K instead of 250K (5x cheaper, still useful).

## 4. Evaluation harness for fine-tuning

Every fine-tuning experiment must:
1. Be A/B tested against the baseline on a **fixed 200-query dev set**.
2. Report R@1, R@5, R@10, NDCG@10.
3. Be reproducible: `experiments/<exp_id>/config.yaml + train.py + checkpoints/`.
4. Be discardable: if it doesn't beat baseline on the dev set, throw it away and document why in `experiments/<exp_id>/RESULT.md`.

## 5. What we will *not* fine-tune

1. **Whisper / PhoWhisper** - too expensive; stays frozen.
2. **YOLO / Places365 / TransNetV2** - good enough off-the-shelf.
3. **The full Vintern-3B or Qwen2.5-VL** - too risky to break; only the LoRA heads.
4. **Embedding models from scratch** - never.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Synthetic captions don't match real query distribution | Generate from both Qwen-VL and Gemini Flash; sample query styles from real user examples + LSC review |
| LoRA overfits to dev set | Hold out a 20% test set never touched during HP tuning |
| Fine-tune hurts in some query class | Per-class slicing on the dev set; ship only if no class regresses by >2% |
| Vintern license issues | Vintern is Apache-2.0; verified. ColVintern same. |

## 8. Acceptance test

The fully-fine-tuned ensemble must beat the off-the-shelf ensemble by **at least 10% R@1 on the dev set**. Otherwise, ship the off-the-shelf models and use the GPU time elsewhere (planner prompt tuning, UI polish).
