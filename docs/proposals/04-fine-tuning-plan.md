# Proposal 04 - Fine-tuning Plan

> Pre-trained models are great but we will get ~5-15% recall@1 lift by fine-tuning on competition-style data. This proposal describes the cheap-and-fast plan.
>
> **What is reused vs novel:** sections 2.1, 2.2, 2.3 (DreamLIP SigLIP-2 LoRA, ColVintern late-interaction, Vintern-1B cross-encoder) are reproductions of published recipes with our Vietnamese data. The novel additions are 2.4 (DiacriticBERT, C1), 2.5 (learned per-task-type fusion, C2), and 2.6 (planner self-distillation, C4). See [`08-original-contributions.md`](08-original-contributions.md) for the full novelty proposal.

## 1. Philosophy

- **LoRA only**. No full-parameter training, ever (except the small heads in C1 and C2, which train new parameters from scratch but on top of frozen backbones).
- **Synthetic data** generated by frontier VLMs. No human labelling effort.
- **Vietnamese-first** captions and queries.
- **One model at a time**. Don't bundle changes; A/B vs baseline always.
- **Ablate the original contributions independently before stacking** (C1, C2, C4 each have a standalone gate; see SS 2.4-2.6 below).

## 2. Six fine-tuning interventions, in priority order

The first three (SS 2.1-2.3) are reproduced from published recipes. The last three (SS 2.4-2.6) are our original contributions.

### 2.1 DreamLIP-style synthetic-caption fine-tuning of SigLIP-2

**Goal**: lift Vietnamese text->image retrieval on competition data.

**Method**:
1. Sample 50K keyframes from the AIC dataset (stratified by place_label / adl_label).
2. For each keyframe, generate **5 long Vietnamese captions** with Qwen2.5-VL-72B INT4:
   - 1 dense caption (50-100 words)
   - 2 short captions (10-20 words)
   - 2 question-style captions ("What is in this image?")
3. **Sub-caption sampling**: at training time randomly drop sentences from the dense caption to form variable-length queries (DreamLIP recipe).
4. **Grouping loss**: positive pairs include (anchor, dense), (anchor, short), (anchor, question). Hard negatives mined from top-50 SigLIP-2 nearest neighbours of the anchor.
5. **LoRA r=16, alpha=32** on Q,K,V,O of both image and text towers.
6. Train 5 epochs on 50K x 5 = 250K pairs, batch 256, LR 1e-4 cosine. Total: ~6 GPU-hours on A6000.

**Expected lift**: +3-7% R@1 on internal dev set. (DreamLIP paper shows 30M pairs matches CLIP-400M; our 250K is enough for a domain shift.)

**Files**:
- `train/dreamlip_siglip2_lora.py`
- `prompts/caption_generation.txt`

### 2.2 ColVintern late-interaction head for Vietnamese document/OCR pages

**Goal**: lift OCR/transcript retrieval for queries involving Vietnamese named entities and small printed text.

**Method**:
1. Take frames with rich OCR (>= 20 words detected).
2. Use **5CD-AI/ColVintern-1B-v1** as base.
3. Fine-tune the projection head only with **MaxSim contrastive loss** on (OCR-text-query, frame-image) pairs.
4. Queries generated from OCR text via Vintern-3B-beta: "Sinh ra mo^.t ca^u truy va^?n tu+. nhien tie^?ng Vie^.t ma` ngu+o+`i du`ng se~ go~ de^? ti`m ki^?m hi`nh a?nh co' va(n ba?n na`y: '<ocr>'"
5. Train 3 epochs on ~30K pairs. ~3 GPU-hours.

**Expected lift**: +2-4% recall on OCR-heavy queries; bigger lift on document-style frames (slides, signage).

### 2.3 Cross-encoder reranker on Vietnamese queries

**Goal**: replace the off-the-shelf BLIP-2 ITM with a Vietnamese-aware cross-encoder.

**Method**:
1. Take 100K (query, frame) pairs from our internal mock-task practice + DreamLIP synthetic captions.
2. Label positives (matching ground truth or top-3 according to ground truth) vs negatives (sampled from hard-negative pool: top-50 SigLIP-2 NN minus the positive set).
3. Architecture: Vintern-1B-v3.5 backbone + a classification head returning logit -> sigmoid.
4. LoRA r=8 on the language and vision adapters; freeze backbone.
5. Train BCE loss, 2 epochs, batch 32, ~4 GPU-hours.

**Expected lift**: +5-10% NDCG@10 on the dev set vs no-rerank.

### 2.4 DiacriticBERT late-interaction head (C1, ours - primary)

**Goal**: lift OCR/ASR-bridged retrieval when Vietnamese diacritics are corrupted by PhoWhisper or PaddleOCR.

**Method** (summary; full method in [`08-original-contributions.md`](08-original-contributions.md) SS 3):
1. Build a controlled diacritic-noise function: `drop_all_diacritics`, `random_drop_p` with `p ~ Beta(2, 5)`, `tone_swap_p`, `mixed`.
2. Take ~500K clean Vietnamese strings from OCR + ASR + captions; generate 4 noisy variants each -> ~2M positive pairs.
3. Hard negatives: top-50 BGE-M3 neighbours of the clean side minus the positive set.
4. Architecture: frozen BGE-M3 + 2-layer MLP (768 -> 384 -> 384) per side + ColBERT MaxSim. Optional LoRA r=8 on last 4 BGE-M3 layers (ablated).
5. Loss: InfoNCE, temperature 0.05, 7 mined hard negatives per anchor.
6. Train 250K steps batch 256, ~3 GPU-hours on A6000.

**Expected lift**: +2-5% R@1 on the OCR/ASR-bridged dev-set slice.

**Files**:
- `train/diacritic_noise.py` (noise function, fuzz-tested)
- `train/diacritic_bert.py`
- `data/diacritic_pairs.parquet`
- `eval/diacritic_robustness.py`

**Ship gate**: must not regress any slice (incl. clean Vietnamese captions) by >1.5%. If it does, ship the SeaLLMs-v3 query-rewriting fallback instead and document the negative result in `experiments/c1/RESULT.md`.

### 2.5 Per-task-type learned fusion (C2, ours - primary)

**Goal**: replace uniform RRF k=60 with a per-task LambdaRank model that knows when to favour image-text vs OCR vs ASR.

**Method** (summary; full method in [`08-original-contributions.md`](08-original-contributions.md) SS 4):
1. For each task type t in {KIS, QA, TRAKE, Ad-hoc}, define weights and a per-ranker score-normalisation function (min-max, z-score, rank-reciprocal).
2. Train LightGBM `LGBMRanker` per task type with LambdaRank objective and NDCG@10 gain. ~30 leaves, 200 trees, early-stop on 20% val split.
3. Training data: the 300-query internal dev set + adversarial queries from proposal 05 SS 3.
4. Train budget: ~30 minutes on CPU. Yes, CPU.

**Expected lift**: +1-3% NDCG@10 over RRF k=60 on the dev set.

**Files**:
- `train/learned_fusion.py`
- `src/retrieval/fusion.py` (runtime: load model, apply, fallback to RRF)
- `eval/fusion_ablation.py`

**Ship gate**: per-task-type model must beat RRF by >=1 point NDCG@10 with non-overlapping bootstrap CI (1000 resamples). Runtime guardrail auto-falls back to RRF if learned model regresses on a streaming 50-query window.

### 2.6 Planner self-distillation via DSPy on operator traces (C4, ours - primary)

**Goal**: train the automatic-track planner on the interactive-track operator's own correct, fast decisions.

**Method** (summary; full method in [`08-original-contributions.md`](08-original-contributions.md) SS 6):
1. Operator-trace logging (proposal 01 SS 5.12) is on from Phase 1. By end of Phase 2 we have ~500-2000 high-quality (query, plan, outcome) triples.
2. Filter to sessions where `final_submission` was correct AND `time_to_submit` was below the team median.
3. Run **DSPy MIPRO** or **BootstrapFewShot** to optimise the planner prompt against tool-choice F1 + downstream R@1 joint objective.
4. At inference, the distilled prompt is paired with kNN retrieval of 3 nearest past (query, plan) examples as in-context demonstrations.
5. Refresh after each preliminary round and each mock-finals batch.

**Expected lift**: +10-20% automatic-track R@1 vs zero-shot SeaLLMs-v3 planner.

**Training budget**: ~1 hour per DSPy round on a single A6000.

**Files**:
- `train/planner_distill.py` (DSPy harness)
- `data/operator_traces.parquet` (rolling log)
- `src/agent/planner_prompt.py`
- `eval/agent_automatic_ablation.py`

**Ship gate**: every distilled prompt is A/B'd on a frozen 100-query held-out set. If the new prompt does not beat the previous by >=2% R@1, the previous prompt is kept.

## 3. Data generation budget

| Asset | Volume | Cost |
|---|---|---|
| Captions (Qwen-VL-72B INT4 local) | 250K @ ~1500 tokens out | 0 (own GPU, ~24 hours on A6000) |
| Cross-encoder labels | 100K pairs | 0 (auto from dev set) |
| Eval queries | 500 | Manual; 1 weekend of team time |

If we lack the GPU for local 72B, fallback to **Gemini 2.5 Flash** at $0.50/M in + $3.00/M out:
- 250K captions x avg (200 in + 1500 out) tokens = 50M in + 375M out
- Cost: $25 in + $1125 out = **~$1150**

Acceptable one-off cost. If budget is tight, do 50K instead of 250K (5x cheaper, still useful).

## 4. Evaluation harness for fine-tuning

Every fine-tuning experiment must:
1. Be A/B tested against the baseline on a **fixed 200-query dev set**.
2. Report R@1, R@5, R@10, NDCG@10.
3. Be reproducible: `experiments/<exp_id>/config.yaml + train.py + checkpoints/`.
4. Be discardable: if it doesn't beat baseline on the dev set, throw it away and document why in `experiments/<exp_id>/RESULT.md`.

## 5. What we will *not* fine-tune

1. **Whisper / PhoWhisper** - too expensive; stays frozen.
2. **YOLO / Places365 / TransNetV2** - good enough off-the-shelf.
3. **The full Vintern-3B or Qwen2.5-VL** - too risky to break; only the LoRA heads.
4. **Embedding *backbones* from scratch** - never. (Note: C1 DiacriticBERT trains a small 2-layer projection head from scratch *on top of a frozen BGE-M3 backbone*; this is not a backbone train and is in scope.)

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Synthetic captions don't match real query distribution | Generate from both Qwen-VL and Gemini Flash; sample query styles from real user examples + LSC review |
| LoRA overfits to dev set | Hold out a 20% test set never touched during HP tuning |
| Fine-tune hurts in some query class | Per-class slicing on the dev set; ship only if no class regresses by >2% |
| Vintern license issues | Vintern is Apache-2.0; verified. ColVintern same. |
| **C1 noise schedule does not match real diacritic-corruption distribution** | Validate the noise function against a 200-sample sanity slice of real PhoWhisper / PaddleOCR outputs; tune `Beta(2,5)` parameters if the mass is misaligned |
| **C2 LightGBM overfits the 300-query dev set** | Leave-one-task-out CV at training time (gap <3 pts); runtime auto-fallback to RRF on streaming-window regression |
| **C4 distilled prompt regresses on real prelim queries unlike Phase-2 mock queries** | Frozen 100-query held-out A/B gate before each prompt replacement; previous prompt retained on regression |

## 8. Acceptance test

The fully-fine-tuned ensemble (reproduced recipes + shipped original contributions) must beat the off-the-shelf ensemble by **at least 10% R@1 on the dev set**, AND at least 2 of {C1, C2, C4} must pass their individual ship gates (SS 2.4-2.6). Otherwise, ship the off-the-shelf models for the failed recipes and use the GPU time elsewhere (planner prompt tuning, UI polish, operator drills).
